# dev-knowledge-base

> 把踩过的坑、修过的 bug、配过的环境，
> 慢慢封印成一套人和 AI 都能直接使用的知识卷轴。

这是一个面向个人使用，也面向 AI agent 协作的开发知识库。

这里主要记录：

- 开发环境配置
- 工具链使用经验
- 常见问题排查过程
- 已验证可复用的解决方案

这个仓库的目标不只是“以后自己回来看得懂”，还包括：

- 少重复踩坑
- 少临时百度
- 少在“我之前明明解决过”里反复轮回
- 让下一个接手问题的人或 AI agent，读完文档就能直接开始处理

## 这个仓库是干什么的

这里记录的不是“随手一记就结束”的零散笔记，而是已经实际验证过的内容，比如：

- 某个问题的稳定解决方案
- 某个工具在特定环境下的可行配置
- 某条排障路径为什么有效
- 某个桌面工具、终端、代理或编码问题该如何判断

希望达到的效果是：

1. 人可以快速看懂
2. AI 可以直接读取
3. AI 读取后，能据此继续排查、执行或修复

比如这次 Windows 下 Codex 中文乱码的问题，一旦文档写清楚，后面再遇到同类问题，就可以直接让 Codex 读取文档，再按文档里的步骤自行尝试解决。

## 收录范围

- 开发环境配置和坑位
- 常见排障结论
- 工具链使用经验
- 可复用的命令与验证步骤

## 记录原则

- 先写成文档，优先解决“下次人和 AI 能不能快速看懂”
- 结论尽量写完整，最好做到复制命令就能复现
- 一篇文档只解决一个核心问题，避免越写越散
- 尽量把判断依据、验证命令和结论一起写清楚，方便 AI 继续往下执行

## 什么时候再进化成 skill

当某个问题开始重复出现，而且你希望 Codex 以后能按固定流程自动执行排查或处理时，再把它从“知识卷轴”升级成 skill 会更合适。

换句话说：

- 一次性结论，先写文档
- 高频重复动作，再做 skill
- 文档负责沉淀知识，skill 负责自动执行流程
